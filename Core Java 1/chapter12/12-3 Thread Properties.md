
## Interrupting Threads

이전 장에서 배운내용에 의하면 과거에는 다른 스레드를 강제로 죽일 수 있었다. 하지만 이는 작업 중인 데이터를 망가뜨리는 부작용이 있다.

그래서 자바는 스레드 종료에 대해 협력적 취소라는 철학을 도입했다. 즉, 강제로 죽일 수는 없지만, Interrupt(요청)을 할 수 있다. 그러면 해당 스레드가 스스로 안전하게 종료하도록 하는 것이다. 

이 요청을 전달하기 위해 자바의 모든 스레드 내부에 보이지 않는 flag(boolean 타입) 하나씩 존재한다. 이를 Interrupted Status(인터럽트 상태)라 한다.
평소에는 false이고, 다른 스레드가 이 스레드의 `interrupt()` 메서드를 호출하면 true가 된다.

스레드의 flag가 true가 됐을 때, 스레드가 현재 어떤 작업을 하고 있느냐에 따라 알아차리는 방식이 달라진다.
- 스레드가 cpu를 배정받아서 실행중일 때 : 이때는 스스로 확인해야한다.
```java
// 내 깃발이 안 올라갔고(false), 아직 할 일이 남았다면 계속 반복
while (!Thread.currentThread().isInterrupted() && moreWorkToDo) {
    // ...
}
// 누군가 interrupt()를 호출해서 깃발이 true가 되면 while문이 종료됨
```
- 스레드가 sleep, wait 일 때 : 코드를 실행할 수 없으므로 flag를 스스로 확인할 수 없다. 자바는 이를 예외로 해결한다. 잠들어 있는 스레드의 `interrupt()` 메서드가 호출되면, 즉시 `InterruptedException` 예외가 발생하고 스레드를 강제로 깨운다.
	- 잠들어있던 스레드가 일어나면, flag는 다시 false가 된다. 따라서 루프 안에서 `sleep()`을 쓴다면 굳이 `isInterrupted()`로 flag를 확인할 필요가 없다.

교재에서는 실무에서 초보자들이 많이 하는 실수를 밝힌다. 바로 `InterruptedException`을 잡고 아무것도 하지 않는 것이다.
```java
void mySubTask(){
	try{
		Thread.sleep(1000);
	}
	catch(InterruptedException e){
		// ...
	}
}
```
- 이것이 위험한 이유는? 누군가 스레드를 종료하려고 interrupt를 발생시켰는데, 하위 메서드에서 예외를 발생시켜놓고 아무 조치가 없이 무시한다면, 메인 run() 메서드는 종료 요청이 왔는지 모르게 된다. -> 프로그램이 종료되지 않고 좀비처럼 남게된다.
->만약 `catch` 블록에서 딱히 처리할 방법이 없다면, 다음 두 가지 중 하나를 해야 한다.
- flag 재설정. 
```java 
catch(InterruptedException e){
	Thread.currentThread().interrupt();
}
```
- 예외 던지기 : 메서드 선언부에 `throws InterruptedException`을 붙여서 예외 처리 책임을 상위 메서드로 넘긴다.


## Daemon Threads

프로그램을 만들면, 메인 작업과는 별개로 백그라운드에서 지속적으로 일해야하는 스레드들이 있다. 주기적으로 타이머를 울리거나, 오래된 캐시 데이터를 청소하는 작업 등이 이에 해당한다. 
자바에서는 이를 데몬 스레드라 부른다. 

일반 스레드(User Thread)가 하나라도 돌아가고 있다면, JVM은 종료되지 않고 계속 프로그램을 실행한다.
하지만, 데몬 스레드들만 있다면 JVM은 데몬 스레드들을 강제 종료시키고 프로그램을 끝낸다.
- `t.setDaemin(true);` 메서드를 통해 데몬스레드임을 지정하면 된다. 다만, 스레드가 실행된 후에는 일반스레드로 변경할 수 없음에 주의해야한다.

## Thread Names

스레드를 생성하면 자바는 기본적으로 `Thread-0`, `Thread-1` 같은 이름을 붙여준다. 복잡하고 대형인 시스템에서 이런 이름들만 가지고 어떤 스레드에서 에러가 발생했는지 추적하기가 힘들다.
- `t.setName("..");` 처럼 스레드에 임의의 이름을 지어줄 수 있다.
- 이렇게 이름을 지어두면, 스레드 덤프 (Thread Dump, 현재 돌아가는 모든 스레드의 상태를 캡처한 것)를 분석할 때, 쉽게 파악이 가능하다.

## Handlers for Uncaught Exceptions

스레드의 run() 메서드 안에는 컴파일러가 강제하는 예외를 밖으로 던질 수 없도록 설계되어 있다. 
하지만 실행 도중 예상치 못한 런타임 에러가 발생한다면 어떻게 될까?
스레드는 그 즉시 비정상적인 종료를 하게 된다. 이때 이 에러를 잡아줄 catch 블록마저 없다면 에러의 원인조차 모른 채 스레드가 종료된다.

해결책: 예외 핸들러 
`Thread.UncaughtExceptionHandler` 라는 인터페이스를 통해 죽을 스레드의 정리가 가능함.
이 핸들러가 예외를 처리하는 순서는 다음과 같다.
1. 개별 스레드 핸들러 - 해당 스레드에 전용으로 설정된 핸들러가 있는지 확인한다.
2. 스레드 그룹(ThreadGroup) - 스레드가 속한 부모 스레드 그룹에 책임을 넘긴다.
3. 기본 전역 핸들러 - 그룹에서도 처리하지 못했다면, 애플리케이션 전체에 설정된 기본 핸들러를 찾는다. 
4. 시스템 에러 출력 - 위 3개 모두 없다면, 콘솔 창에서 스택 트레이스를 출력하고 스레드를 종료시킨다. 
교재에서는 스레드들을 묶어 관리하는 `ThreadGroup` 클래스의 존재를 설명하면서도, 현재 자바에서는 컬렉션 프레임워크나 스레드 풀 등의 훨씬 더 발전된 관리 도구들이 있기 때문에, 자신의 프로그램에서 `ThreadGroup`을 직접 사용하지 말 것을 권장한다.

---
1. 예외 핸들러 도입 배경 
자바에서 각 스레드는 자신만의 독립적인 Call Stack을 가진다. 메인스레드와 새로운 스레드는 완전히 길이 갈라진다. (`start()`를 호출하게 되면)
예를들어 메인 스레드에서 A라는 스레드를 `start()` 하면, A의 콜 스택의 시작점에는 `run()`이 있기 마련이다. A의 내부에서 런타임 에러가 발생하면, run 메서드 다음으로 던져질 곳이 없다. 그렇기에 이를 방지하기 위해 도입된 것이 예외 핸들러이다.
2. 핸들러의 구현과 연결 방식
예외 핸들러는 외부에서 구현된 객체를 만들어 스레드에 주입하는 방식이다. 
`Thread.UncaughtExceptionHandler`는 `Thread` 클래스 내부에 정의된 중첩 인터페이스이다. 이 인터페이스는 `void uncaughtException(Thread t, Throwable e)` 라는 단 하나의 메서드만 가지고 있다. 
예시
```java
Thread myThread = new Thread(myRunnable);

myThread.setUncaughtExceptionHandler((Thread t, Throwable e) -> {
	System.out.println(t.getName() + " "+ e.getMessage())
});

myThread.start();
```
3. 핸들러의 역할
핸들러의 `uncaughtException`이 호출되는 시점은 사실상 스레드가 죽을 상태이다. 그렇기에 핸들러는 로깅, 알림, 자원 해제 또는 재시작의 역할을 한다.
4. `ThreadGroup`
c언어나 리눅스 환경에서 프로세스 그룹을 통해 여러 프로세스에 한번에 시그널을 보내듯, 스레드를 일괄적으로 통제하기위한 단위이다.
5. 기본 적역 핸들러의 범위
윈도우든 리눅스든 자바 프로그램을 실행하면, OS 입장에서는 JVM이라는 거대한 프로세스가 실행되는 것이다. (스레드의 개수와 상관없이)
그렇기에 기본 전역 핸들러는 하나의 프로세스 JVM 단위이다.

## Thread Priorities

자바의 모든 스레드는 1부터 10 사이의 우선순위를 가진다. 기본값은 5이다. (`NORM_PRIORITY`)
이론상 운영체제의 스케줄러는 우선순위가 높은 스레드에게 cpu를 더 먼저, 많이 할당해 주어야 한다.

교재는 이 기능을 설명하면서 동시에 오늘날에는 이 기능을 절대 사용해서는 안된다고 한다. 그 이유는 자바의 우선순위가 운영체제에 전적으로 의존하기 때문이다. 
- 시스템 의존성 문제: 자바의 우선순위 10단계가 윈도우(우선순위 7단계)로 넘어가면 몇 개의 자바 단계가 윈도우의 같은 단계로 뭉뚱그려진다.
- 플랫폼 무시 문제: 리눅스(Linux)용 오라클 JVM의 경우, 스레드 우선순위를 아예 무시하고 모든 스레드를 완전히 동일하게 취급해버린다.
- 결론: 윈도우에서는 잘 작동하던 우선순위 제어 로직이 리눅스 서버에 올리는 순간 완전히 다르게(혹은 아예 작동하지 않게) 동작하게 된다. 따라서 프로그램의 흐름을 제어할 때 스레드 우선순위에 기대는 것은 위험하다.
